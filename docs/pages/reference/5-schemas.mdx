This page documents how Studyflow uses LinkML as the authoring format for schema definitions.

## Core structure

Studyflow LinkML files follow this top-level layout:

```yaml
id: http://behaverse.org/schemas/studyflow
name: studyflow
prefixes:
  studyflow: http://behaverse.org/schemas/studyflow/
default_prefix: studyflow
enums: {}
types: {}
classes: {}
```

## Classes and attributes

Each class defines a domain element and their attributes. For example:

```yaml
classes:
  StartEvent:
    class_uri: bpmn:StartEvent
    attributes:
      requiresConsent:
        range: boolean
        ifabsent: "false"
        categories: [Privacy]
```

## Extensions

`extensions.moddle` is used to control moddle-specific behavior for a property.

- `redefines`
- `replaces`
- `default`

`extensions.studyflow` is used for modeler/UI behavior.

- `hidden`
- `condition.language`
- `condition.body`

## Examples

Schemas may define top-level `examples` entries that appear in the modeler create/append menus.
Each entry uses `object.type` to identify the schema type to instantiate, plus any default property values to apply.

```yaml
examples:
  - description: Preprocess EEG data for a single subject.
    object:
      type: MapData
      bpmn:name: Preprocess EEG (subject)
      bpmn:documentation: Preprocess EEG data, including filtering, artifact removal, and normalization.
      isDataOperation: true
      icon: iconify icon-park--eeg
```

### Example mixins

`examples[].object.mixins` lets an example compose additional defaults from other types.
This is useful when the example should behave like a specialized BPMN shape without changing the underlying schema class.

```yaml
examples:
  - description: Preprocess EEG data for a single subject.
    object:
      type: MapData
      bpmn:name: Preprocess EEG (subject)
      bpmn:documentation: Preprocess EEG data, including filtering, artifact removal, and normalization.
      isDataOperation: true
      icon: iconify icon-park--eeg
      mixins:
        - bpmn:SubProcess
```

- Use `mixins` for example-time composition, not for defining reusable schema inheritance. Schema inheritance still belongs on classes via `is_a` and class-level `mixins`.

### Nested subprocess flow elements

If an example resolves to `bpmn:SubProcess`, it may also declare `examples[].object.flowElements` to create internal nodes and connections together with the root subprocess.

```yaml
examples:
  - description: Preprocess EEG data for a single subject.
    object:
      type: MapData
      bpmn:name: Preprocess EEG (subject)
      bpmn:width: 420
      bpmn:height: 180
      mixins:
        - bpmn:SubProcess
      flowElements:
        - id: eeg_filter
          type: MapData
          bpmn:name: Filter signal
          x: 70
          y: 65
        - id: eeg_clean
          type: MapData
          bpmn:name: Remove artifacts
          x: 240
          y: 65
        - bpmn:type: bpmn:SequenceFlow
          sourceRef: eeg_filter
          targetRef: eeg_clean
```


Examples may declare promoted attributes explicitly using LinkML-style `attributes:` definitions.

```yaml
examples:
  - description: Preprocess EEG data for a single subject.
    attributes:
      param1:
        range: string
        description: Example-specific promoted attribute.
        ifabsent: "default_value"
    object:
      type: MapData
      bpmn:name: EEGPrep
```

These attributes are not added back onto the original schema class. Instead, the modeler generates an example-scoped derived moddle type for that specific example object, so the extra attributes persist in BPMN XML without changing other `MapData` examples or the base class definition.

If the same example also includes undeclared scalar properties such as `param1: value1`, those are inferred onto the same example-scoped derived type rather than onto the parent schema class.

## Common mistakes

- Prefix mismatch: class or type references using the wrong prefix (for example `studyflow:` vs local names).
- Missing `range`: attributes without range default to string-like behavior and may produce unintended interfaces.
- Invalid extension shape: malformed `extensions.studyflow.condition.body` JSON disables conditional visibility.
- Incorrect `redefines` owner: invalid targets are dropped.
